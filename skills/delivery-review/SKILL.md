# 交付协作工作流（Loop 模式）

> "你不知道远比你知道的要重要的得多。"——塔勒布

**执行入口：`/delivery-review` 命令。** 本文件是方法论与数据模型参考。命令内部已用 `.delivery-review/state.json` 状态机驱动循环，不依赖模型记忆轮次。

---

## 元认知原则

### 边界感知三层

| 层级 | 含义 | 必须输出 |
|------|------|---------|
| 已知 | 确定检查过，结论明确 | ✅ 确认 / ❌ 有问题 |
| 不确定 | 检查了但没把握 | ⚠️ + 原因 |
| 未检查 | 根本没覆盖到的维度 | 🔍 列出（最重要） |

**每个交付节点必须输出「未知清单」——列出 AI 无法确认的风险点。这个清单比检查清单更重要。**

---

## 角色与隔离

| 角色 | 执行者 | 权限 | 职责 |
|------|--------|------|------|
| 修复工程师 | `delivery-fixer` 子代理 | 读写代码 | 改代码、自检、处理 risk |
| 资深审查者 | `delivery-reviewer` 子代理 | **只读（工具层禁止写）** | 独立审查、报 risk、打分、确认收敛 |
| 编排者 | 主 Agent / `/delivery-review` | 读写 + spawn | 确认上下文、输出方案与目标、驱动 loop、裁决、人工验收 |

> 审查者的「只读」由 agent 定义的 `disallowedTools` 在工具层强制，不靠 prompt 请求。

## 通信机制与数据模型

### 存储

- 对话记录：`git notes --ref=delivery-review`
- 循环状态（机器判定）：`.delivery-review/state.json`
  ```json
  {"round":0,"consecutive_clean":0,"last_score":0,"exit_reason":"","max_rounds":3}
  ```

### 留言格式

```
<ID>|<from>|<type>|<reply_to>|<正文>
```

| 字段 | 说明 |
|------|------|
| ID | 递增整数，当前最大 ID + 1 |
| from | claude / reviewer / system |
| type | context / review / risk / done / score / approve / dispute / confirm / withdraw |
| reply_to | 回复的留言 ID，无则 0 |
| 正文 | 不含竖线 `|`（需转义为 `\|`） |

### type 枚举

| type | 谁发 | 含义 |
|------|------|------|
| `context` | 编排者 | 目标 + 背景 + 方案 |
| `review` | 修复方 | 改动说明 + 意图声明 + 自检 + 未知清单 |
| `risk` | 审查方 | 问题描述 + 分级 + 质量分 |
| `done` | 修复方 | 对某条 risk 的处理结果 |
| `score` | 审查方 | 本轮质量评分 + 收敛判断 |
| `approve` | 审查方 | 所有 risk 已解决，建议人工验收 |
| `dispute` | 修复方 | 对某条 risk 的反驳（附理由） |
| `confirm` | 审查方 | 收到 dispute 后自我核查结果（仍坚持） |
| `withdraw` | 审查方 | 撤回原 risk（误判） |

### 风险分级

| 级别 | 含义 | 处理要求 |
|------|------|---------|
| P0-阻塞 | 安全漏洞、数据丢失、功能不可用 | 必须修复才能退出 loop |
| P1-重要 | 逻辑错误、性能退化、边界缺失 | 必须修复或说明不修复理由 |
| P2-建议 | 代码风格、命名、可维护性 | 可 defer，人工验收时酌情处理 |

---

## Loop 退出条件（由 state.json 状态机判定，优先级从高到低）

1. **oscillation / dispute 未解** → 强制退出，移交人工裁决
2. **converged**：连续两轮 `risk==0` 或 `新增 risk==0` → 理想退出
3. **quality_gate**：`质量分 >= 7` 且无 P0/P1 → 次优退出
4. **max_rounds**：达到 `max_rounds`（默认 3）→ 兜底退出

---

## Dispute 机制

```
审查方发 risk
  ↓
修复方不同意 → dispute（反驳理由 + 证据）
  ↓
审查方自我核查
  ├─ 仍认为有问题 → confirm → 退出 loop，人工裁决
  └─ 发现误判   → withdraw → 继续 loop
```

---

## 人工验收（最后一道防线，不可跳过、不可由 AI 代理）

| 范围 | 内容 | 必须/可选 |
|------|------|----------|
| 改动点 | 修复方修改过的代码对应功能 | 必须 |
| 未解决项 | loop 中未解决的 risk | 必须 |
| 双方未知清单 | 修复方与审查方都无法确认的风险 | 必须 |
| 震荡点 | 反复修改的部分 | 必须 |
| 回归测试 | 原有功能是否被破坏 | 可选 |
| 体验检查 | 操作顺畅度、提示清晰度 | 可选 |

## 附：塔勒布元认知速查

1. **我知道什么？** —— 确定检查过的
2. **我不确定什么？** —— 检查了但没把握的
3. **我不知道什么？** —— 根本没想到要检查的（最重要）
4. **如果我错了，后果是什么？** —— 下行风险不对称性
5. **这个改动重复一万次，会在某一次崩溃吗？** —— 遍历性检验

> 火鸡在被杀前的每一天都通过了「自检清单」。未知清单才是真正的防线。
